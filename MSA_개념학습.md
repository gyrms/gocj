## 1. 기본 개념

### MSA vs 모놀리식 아키텍처

| | 모놀리식 | MSA |
|--|---------|-----|
| 구조 | 하나의 애플리케이션에 모든 기능 | 기능별로 독립적인 서비스로 분리 |
| 배포 | 전체를 한 번에 배포 | 서비스별 독립 배포 가능 |
| 장애 | 하나 터지면 전체 영향 | 장애 격리 가능 |
| 개발 | 초기엔 빠름 | 초기 구축 복잡 |
| DB | 하나의 DB 공유 | 서비스별 독립 DB |

### MSA 장점
- 서비스별 **독립 배포** 가능 → 배포 속도 빠름
- **장애 격리** → 하나의 서비스가 죽어도 전체 시스템 영향 최소화
- 서비스별 **독립적인 기술 스택** 선택 가능
- **트래픽에 따른 서비스별 스케일 아웃** 가능

### MSA 단점
- 서비스 간 통신 복잡도 증가
- 분산 트랜잭션 처리 어려움
- 모니터링/운영 복잡도 증가
- 초기 인프라 구축 비용 큼

### 언제 MSA가 필요한가
- 서비스 규모가 크고 팀이 여러 개일 때
- 특정 기능의 트래픽이 급격히 증가할 때 (e.g. 이벤트 시 상품 서비스만 스케일 아웃)
- 독립적인 배포 주기가 필요할 때

### 언제 모놀리식이 나은가
- 초기 스타트업, 작은 팀
- 서비스 도메인이 아직 명확하지 않을 때
- 빠른 프로토타이핑이 필요할 때

## 2. 서비스 분리

### 도메인 기반 서비스 분리 방법
- **비즈니스 도메인 기준**으로 분리 (기술 기준 X)
- 서비스 간 결합도를 최소화하도록 분리
- 각 서비스는 자신의 DB를 소유 (DB 공유 금지)

### 바운디드 컨텍스트 (Bounded Context)
- DDD(도메인 주도 설계)에서 나온 개념
- **하나의 도메인 모델이 유효한 경계**를 의미
- 같은 단어도 컨텍스트마다 의미가 다를 수 있음
  - 예) "상품" → 상품 서비스에서는 재고/가격, 주문 서비스에서는 주문된 상품 정보
- MSA에서 서비스 경계 = 바운디드 컨텍스트 경계

### 올리브영 기준 서비스 분리 예시
```
회원 서비스    - 회원가입, 로그인, 회원정보
상품 서비스    - 상품 등록, 조회, 재고 관리
주문 서비스    - 주문 생성, 주문 내역
결제 서비스    - 결제 처리, 취소, 환불
배송 서비스    - 배송 상태 추적, 배송지 관리
쿠폰 서비스    - 쿠폰 발급, 적용, 만료
리뷰 서비스    - 리뷰 작성, 조회
```

## 3. 서비스 간 통신

### 동기 통신 vs 비동기 통신

| | 동기 (REST, gRPC) | 비동기 (Kafka, RabbitMQ) |
|--|------------------|------------------------|
| 응답 | 즉시 응답 필요 | 응답 기다리지 않음 |
| 결합도 | 높음 | 낮음 |
| 장애 전파 | 호출 대상 장애 시 함께 영향 | 장애 격리 가능 |
| 사용 예 | 실시간 조회, 인증 | 주문 완료 후 알림, 재고 차감 |

### 언제 동기를 쓰는가
- 결과를 즉시 알아야 할 때 (결제 승인 여부)
- 사용자가 응답을 기다려야 할 때

### 언제 비동기를 쓰는가
- 즉시 응답이 필요 없을 때 (주문 완료 후 이메일 발송)
- 장애 격리가 중요할 때
- 대용량 이벤트 처리 (재고 차감, 통계)

### API Gateway
- 클라이언트와 서비스 사이의 **단일 진입점**
- 역할:
  - 인증/인가 처리 (토큰 검증)
  - 라우팅 (요청을 적절한 서비스로 전달)
  - 로드 밸런싱
  - Rate Limiting (요청 제한)
  - 로깅/모니터링

## 4. 장애 처리

### 서킷 브레이커 (Circuit Breaker)
- 연속 실패가 일정 횟수 이상이면 **호출 자체를 차단**하는 패턴
- 전기 회로 차단기에서 이름을 따옴

```
상태 3가지:
Closed   → 정상 동작 (요청 통과)
Open     → 장애 감지 (요청 차단, 바로 에러 반환)
Half-Open → 일부 요청만 허용하며 복구 확인
```

- 대표 라이브러리: **Resilience4j**
- 장점: 장애 서비스로의 요청을 막아 **연쇄 장애(Cascade Failure) 방지**

### 분산 트랜잭션 문제
- MSA에서 각 서비스가 **별도 DB**를 가지기 때문에 하나의 트랜잭션으로 묶을 수 없음
- 예) 주문 생성 + 재고 차감 + 결제 처리 → 3개의 서비스, 3개의 DB
  - 결제는 성공했는데 재고 차감 실패 → 데이터 불일치

### Saga 패턴
분산 트랜잭션을 **여러 개의 로컬 트랜잭션**으로 나눠 처리하는 패턴

#### Choreography (코레오그래피)
- 서비스들이 **이벤트로 서로 소통** (중앙 조율자 없음)
- 각 서비스가 이벤트를 발행하고 구독

```
주문서비스 → 주문생성이벤트 발행
  → 재고서비스 구독 → 재고차감 → 재고차감완료이벤트 발행
    → 결제서비스 구독 → 결제처리
```

- 장점: 서비스 간 결합도 낮음
- 단점: 전체 흐름 파악 어려움

#### Orchestration (오케스트레이션)
- **중앙 조율자(Saga Orchestrator)**가 흐름을 관리
- Orchestrator가 각 서비스에 직접 명령

```
Orchestrator → 재고서비스 호출 → 결제서비스 호출 → 완료
실패 시 → 보상 트랜잭션 실행 (재고 복구, 결제 취소)
```

- 장점: 전체 흐름 한눈에 파악 가능
- 단점: Orchestrator에 로직이 집중됨

## 5. 기타

### 서비스 디스커버리
- MSA에서 서비스 인스턴스의 IP/Port가 동적으로 바뀜
- **서비스 디스커버리** = 서비스의 위치(IP/Port)를 자동으로 찾아주는 메커니즘
- 대표 도구: **Eureka** (Netflix), Consul
- 동작:
  - 서비스 시작 시 레지스트리에 자신을 등록
  - 호출할 때 레지스트리에서 대상 서비스 주소 조회

### 모놀리식 → MSA 전환 전략
- **Strangler Fig 패턴** (가장 일반적)
  - 기존 모놀리식을 유지하면서 기능을 하나씩 분리
  - 점진적으로 모놀리식을 줄여나감
- 한 번에 전체를 바꾸는 Big Bang 방식은 위험 → 권장하지 않음

## 면접 자주 나오는 질문

---

### Q1. MSA와 모놀리식의 차이를 설명해주세요.

**모놀리식**은 회원, 상품, 주문, 결제 등 모든 기능이 하나의 애플리케이션 안에 존재하고 하나의 DB를 공유하는 구조입니다.

**MSA**는 기능을 독립적인 서비스 단위로 분리하고, 각 서비스가 독립된 DB와 독립된 배포 주기를 갖는 구조입니다.

모놀리식은 초기 개발이 빠르고 단순하지만, 서비스가 커질수록 특정 기능의 장애가 전체 시스템에 영향을 주고, 트래픽이 몰리는 기능만 선택적으로 스케일 아웃하기 어렵습니다.

MSA는 장애 격리가 가능하고 서비스별 독립 배포와 스케일 아웃이 가능하지만, 서비스 간 통신 복잡도가 올라가고 분산 트랜잭션 처리가 어려워지는 단점이 있습니다.

---

### Q2. MSA 환경에서 분산 트랜잭션은 어떻게 처리하나요?

MSA에서는 각 서비스가 별도 DB를 소유하기 때문에, 주문 생성 / 재고 차감 / 결제 처리를 하나의 트랜잭션으로 묶을 수 없습니다.

이를 해결하기 위해 **Saga 패턴**을 사용합니다. Saga 패턴은 각 서비스의 로컬 트랜잭션을 순서대로 실행하고, 중간에 실패하면 이전 단계를 취소하는 **보상 트랜잭션(Compensating Transaction)**을 실행하여 데이터 정합성을 맞춥니다.

예를 들어 결제 실패 시, 이미 차감된 재고를 복구하고 생성된 주문을 취소하는 보상 트랜잭션이 실행됩니다.

구현 방식은 서비스들이 이벤트로 자율 소통하는 **Choreography**와, 중앙 Orchestrator가 흐름을 관리하는 **Orchestration** 두 가지가 있습니다.

### Q3. 서비스 간 통신 방법에 대해 설명해주세요.

크게 **동기 통신**과 **비동기 통신**으로 나뉩니다.

**동기 통신(REST, gRPC)**은 요청 후 즉시 응답을 받아야 하는 경우에 사용합니다. 결제 승인 여부나 실시간 재고 조회처럼 사용자가 결과를 즉시 알아야 할 때 적합합니다. 단, 호출 대상 서비스에 장애가 생기면 함께 영향을 받는 단점이 있습니다.

**비동기 통신(Kafka, RabbitMQ)**은 응답을 기다리지 않고 이벤트를 발행하는 방식입니다. 주문 완료 후 알림 발송, 배송 상태 업데이트처럼 즉시 응답이 필요 없는 경우에 사용합니다. 서비스 간 결합도가 낮아지고 장애 격리가 가능한 장점이 있습니다.

### Q4. 서킷 브레이커 패턴이 무엇인가요?

특정 서비스에 연속적으로 장애가 발생할 때, 해당 서비스로의 호출 자체를 차단하는 패턴입니다. 전기 회로의 차단기에서 이름을 따왔습니다.

상태는 3가지로 동작합니다.
- **Closed**: 정상 상태, 요청이 통과됩니다.
- **Open**: 장애 감지, 요청을 차단하고 즉시 에러를 반환합니다.
- **Half-Open**: 일부 요청만 허용하여 서비스 복구 여부를 확인합니다.

이를 통해 장애 서비스로의 요청이 쌓이는 것을 막아 **연쇄 장애(Cascade Failure)**를 방지하고 시스템 전체 안정성을 높입니다. Java에서는 **Resilience4j** 라이브러리를 주로 사용합니다.

### Q5. API Gateway의 역할은 무엇인가요?

API Gateway는 클라이언트와 여러 서비스 사이의 **단일 진입점** 역할을 합니다.

주요 역할은 다음과 같습니다.
- **인증/인가**: JWT 토큰 검증을 Gateway에서 처리하여 각 서비스가 중복 구현하지 않아도 됨
- **라우팅**: 요청 URL에 따라 적절한 서비스로 전달
- **로드 밸런싱**: 트래픽을 여러 인스턴스에 분산
- **Rate Limiting**: 특정 클라이언트의 과도한 요청 제한
- **로깅/모니터링**: 요청/응답 통합 관리

API Gateway가 없으면 클라이언트가 각 서비스의 주소를 직접 알아야 하고, 인증 로직을 모든 서비스에 중복 구현해야 하는 문제가 생깁니다.

### Q6. 올리브영 서비스를 MSA로 나눈다면 어떻게 하겠습니까?

비즈니스 도메인 기준으로 **회원 / 상품 / 주문 / 결제 / 배송 / 쿠폰 / 리뷰** 서비스로 분리하겠습니다.

각 서비스는 독립된 DB를 소유하고, 서비스 간 직접 DB 접근은 금지합니다.

**통신 전략**은 다음과 같이 적용하겠습니다.
- 주문 시 결제 승인 여부 → 동기 통신 (즉시 결과 필요)
- 주문 완료 후 재고 차감, 배송 시작, 쿠폰 사용 처리 → 비동기 통신 (Kafka 이벤트)
- 주문 / 재고 차감 / 결제의 분산 트랜잭션 → Saga 패턴으로 처리, 실패 시 보상 트랜잭션 실행

**스케일 아웃 전략**은 이벤트나 세일 기간에 트래픽이 집중되는 상품 조회 서비스와 주문 서비스를 독립적으로 스케일 아웃하겠습니다.
